## Repository snapshot (at 2026-07-13)

- **Monorepo** (uv workspace, [ADR-001]): `packages/{core,router,retrieval,tools,runtime,trace}` + `apps/{cli,gateway}`.
- **Gateway** (`apps/gateway`): FastAPI, OpenAI-compatible `/v1/chat/completions` + `/v1/models`, `/health`, `/health/ready`. Stateless. Bearer-key auth.
- **Runtime** (`packages/runtime/agent.py`): a single `AgentRuntime` loop — retrieve → complete → execute tools → repeat, both streaming and non-streaming.
- **Router** (`packages/router`): Ollama + OpenAI clients; `EscalationPolicy` (binary local→frontier on error/empty).
- **Retrieval** (`packages/retrieval`): SQLite FTS5 index, git-aware incremental watch ([ADR-014]), naive char/line chunker.
- **Tools** (`packages/tools`): exactly two — `read_file`, `ripgrep` — hardcoded in `registry.py`, sandbox-path-checked in `executor.py`.
- **Trace** (`packages/trace`): SQLite span store; CLI `annulus traces list|show|last` (the just-merged tooling).
- **Config** (`packages/core/config.py`, `config/*.yaml`): pydantic settings + `default.yaml` + `models.yaml` (5 profiles, capability flags `supports_tools`/`expose_reasoning`).
- **15 ADRs**: 001–005 + 014 **Accepted**; 006–013, 015 **Proposed** (011 deliberately open-ended).

Everything scientific, multi-agent, skill-based, plan-based, permission-based, embedding-based, and graph-based is **documentation only** — no code path exists yet.

## Pillar 1 — Single agent runtime, multiple interfaces

**Alignment**
- The core boundary is real and clean: agent logic lives only in `AgentRuntime` (`packages/runtime/agent.py`); clients never implement tool loops or retrieval injection. Explicitly codified in [ADR-006] ("Agent logic stays in `AgentRuntime`; clients do not implement tool loops").
- One entry point: gateway `/v1` ([ADR-003]). CLI (`apps/cli/main.py`) and Continue are both OpenAI-compatible clients of the same engine.
- Dependency singletons wired once in `apps/gateway/deps.py` (`init_dependencies`) — one runtime instance owns router/retriever/tools/trace.
- Layered vision diagram in `vision.md` matches this ("Experience layer → Annulus engine → workspace stores → compute tier").

**Gaps**
- The pillar wants a **persistent daemon/sidecar that "owns agents, tools, context, permissions, routing."** Today it owns tools, context, and routing — but **not agents (plural)** and **not permissions** (neither concept exists in code).
- No **daemon lifecycle / "latch"** — the gateway is a plain uvicorn process; ADR-012 (target-native sidecar `annulus serve`) is Proposed, unbuilt.
- The **IDE interface is generic Continue**, not the described Cursor-like "Agents window / repo-aware inline edits." That surface is v1.0 UI ([ADR-006]) — unstarted.
- The **CLI is not yet Claude-Code-like**: it exposes only `health/index/chat/traces`. No plans, no commands, no deep tool orchestration.
- The future engine APIs that would make "multiple interfaces on one runtime" real — `GET /v1/traces/{id}`, `GET /v1/index/status`, `POST /v1/agents/run` — are listed in [ADR-006] but **none are implemented** (only the `X-Annulus-Trace-Id` header + CLI reading the DB directly).

## Pillar 2 — Local-first context substrate

**Alignment**
- Strong principle discipline: `vision.md` "Sidecar, not embed"; all state under `.annulus/` on the mounted workspace (`index.db`, `traces.db`, `index_meta.json`). Env-driven `ANNULUS_WORKSPACE_ROOT` / `ANNULUS_DATA_DIR`.
- Working local FTS5 substrate ([ADR-004]) with **git-aware incremental refresh** ([ADR-014], `git_delta.py`, `meta.py`) — a genuinely well-engineered piece (measured 5-min full-index pain motivated it).
- Partial inspectability: `/health` returns `index: {files, chunks}`; `annulus traces show` surfaces retrieval hits per run.
- Forward path documented: embeddings ([ADR-009]) and GraphRAG-lite ([ADR-010]) explicitly targeted at the **same `.annulus/` store**.

**Gaps**
- The substrate is **lexical-only**. Embeddings, graph, **project/repo memory**, **experiment state**, and **provenance** — all named in the pillar — are Proposed or absent. `grep provenance` = 0 hits.
- **No formal "latch"/attach lifecycle** — indexing is a manual/CLI/gateway-background task; ADR-012 attach model is Proposed.
- **Context-store schema is undocumented** except by reading `store.py`. No ADR documents the substrate schema or its evolution to embeddings/graph/memory tables. Inspection is ad-hoc (CLI + health counts); the `GET /v1/index/status` inspection API is unbuilt.
- **ZDR is only discussed for frontier *inference*, never for *context*.** The pillar wants "compliant ZDR endpoints" for the context substrate — no such story or config exists.
- **Chunking is naive** (char/line windows in `indexer._chunk_text`); AST-aware code splitting is a noted follow-up ([ADR-009]) and there is no non-code chunking (notebooks, data, bib) — relevant to Pillar 8.
- Traces are explicitly **"not chat memory"** ([ADR-004], README) and there is **no memory store** — [ADR-011] defers it to v1.1+.

## Pillar 3 — Policy-driven model routing and escalation

**Alignment**
- `ModelRouter` + `EscalationPolicy` (`packages/router/{router,escalation}.py`) implement local-first with deliberate escalation ([ADR-005]).
- Profiles are declarative in `config/models.yaml`; escalation is toggleable and recorded (`annulus.escalated`, trace spans).
- [ADR-007] articulates a **tiered** ladder (local → lab-gpu → frontier) and remote OpenAI-compatible/Ollama endpoints as first-class compute tiers — aligned with the "user-controlled, not same-machine" reading of local-first.

**Gaps**
- Escalation is **binary and reactive**, not policy-driven: `EscalationPolicy.should_escalate` fires only on **local error** or **empty response** (`escalation.py`). There is **no confidence signal** anywhere, so "escalate only when local confidence is low" is unimplementable today.
- **No per-project / per-agent policy object.** Escalation config is global (`models.escalation`), and routing is just "client sends a `model` string or falls to `default_profile`" (`router.resolve_profile`). "This project allows ZDR/frontier" cannot be expressed.
- [ADR-007]'s tiered ladder is **Proposed only**: `_escalate` hard-jumps to the single `frontier_profile`; there is no multi-tier walk.
- **No data-egress enforcement.** Given the ITAR framing, nothing prevents workspace content (retrieved chunks, tool output) from being sent to a frontier endpoint during escalation. [ADR-005]/[ADR-007] mention security but enforce nothing, and **no ADR addresses egress/compliance policy** — a significant gap for the stated NASA/ITAR use case.

## Pillar 4 — Clear separation of primitives: TOOLS vs SKILLS vs AGENTS

**Alignment**
- **Tools are first-class**: schemas in `registry.py`, execution/sandboxing in `executor.py`, gated by `profile.supports_tools` ([ADR-004]).
- **Agents-as-orchestrators** are conceptually described in [ADR-008] (orchestrator + research/code/verify workers).

**Gaps (this is the weakest-covered structural pillar)**
- **SKILLS do not exist as a primitive.** The word appears **only** in [ADR-011] as a future ".annulus/ skills" idea. There is no skills package, no skills directory, no schema, no loader, and **no ADR defining skills** or the tool↔skill↔agent boundary. The pillar's "skills directory similar to Claude/OpenClaw skills" is entirely unrealized.
- **AGENTS are singular and hardcoded.** There is exactly one `AgentRuntime`. No notion of an *agent* as a configurable entity with **role, prompt, memory scope, permissions** (planner/executor/critic/domain-specialist). [ADR-008] describes a swarm but never defines what an individual agent *is* as a config object — so the swarm ADR floats above a missing primitive.
- **Tools are not extensible**: two hardcoded functions, no plugin/registration mechanism, no MCP (deferred to v0.6). No scientific tools (simulation runners, data loaders, notebook execution).
- Net: the intended **3-tier taxonomy is ~1.5 tiers** (tools + one monolithic agent). The separation the pillar demands is aspirational.

## Pillar 5 — Plan-before-act interaction model

**Alignment**
- Essentially none in code. The closest is [ADR-008]'s "orchestrator decomposes goal into a structured plan" — Proposed, future.
- Execution *logging* exists (traces), partially satisfying the "execution is logged" clause.

**Gaps**
- `grep -i "plan mode|planning"` = **0 hits.** There is **no plan proposal, no approve/modify/reject flow, and no ADR** covering plan-before-act as an interaction model.
- `AgentRuntime` runs **fully autonomously** up to `max_iterations` (default 8), executing tool calls with **no human checkpoint**. For the high-stakes / ITAR / scientific framing, the absence of an approval gate is a notable risk.
- "Reversible where possible" is untested because current tools are read-only; but `propose_edit` (v0.4, [ADR-015]) will make this urgent and there is no reversibility/checkpoint primitive designed.

## Pillar 6 — Dual control planes (IDE + CLI)

**Alignment**
- Both planes exist and hit the same engine: CLI (`annulus health/index/chat/traces`) and IDE via Continue (`docs/continue-config.example.yaml`, [ADR-006]).
- **Logs are genuinely first-class in both** (trace store + `X-Annulus-Trace-Id` header + `annulus traces` CLI).

**Gaps**
- Of the pillar's "permissions, plans, logs first-class in both," **only logs are delivered.** Permissions: absent. Plans: absent.
- IDE plane = **generic Continue chat**, not the Cursor-like "repo-aware Agents window with fast inline edits." Inline edit/apply (`propose_edit` + Continue apply role) is v0.4 Proposed ([ADR-015]), unbuilt. Note: the vision lists "git tools" in v0.3, but `registry.py` still has **only `read_file` + `ripgrep`** — `git_status`/`git_diff` are **not present**, so the v0.3 "git tools" item appears incomplete in this tree.
- CLI plane is thin: no plan commands, no eval commands, no large-refactor/research orchestration, no external-tool orchestration (Wolfram / notebooks / simulators).
- No notebook client (Pillar 8 overlap).

## Pillar 7 — Evaluation, logging, provenance as core runtime features

**Alignment (strongest-implemented pillar)**
- **First-class span tracing**: `TraceStore` records `chat.completions`, `retrieval.search`, `agent.iteration`, `tool.*` with timing/status/attributes. Wired throughout `agent.py` and `chat.py`.
- **Trace CLI** just merged (`apps/cli/traces.py`): `list/show/last`, JSON output, `X-Annulus-Trace-Id` correlation. `build_span_tree` already supports nesting.
- **Eval devcontainer** (`.devcontainer/eval/`) + **tool-compat probe** (`scripts/probe_ollama_tools.py`) + hand-maintained matrix ([ADR-013]) = a real eval discipline.
- `annulus` metadata block (escalated / retrieval_hits / tool_calls / iterations) on non-streaming responses.

**Gaps**
- **Provenance is not actually built.** Traces are explicitly *observability, not memory/provenance*. There is no record linking **inputs → data used → model → output** for reproducibility or SFT/RL export. `grep provenance` = 0.
- **No eval suite / regression harness is committed** — eval is manual dogfooding + a probe script. The "eval runbook" and prompt matrices are Proposed (v0.4-C, [ADR-015]).
- **Trace tree is flat**: `parent_span_id` exists in the schema and `build_span_tree`/`_print_span_node` support hierarchy, but **the runtime never sets `parent_span_id`** (README: "planned once the runtime wires `parent_span_id`"). So multi-agent parent/child provenance ([ADR-008]) is unsupported, and the nesting code is currently dead.
- **Trace attributes truncate content** (`query[:200]`, hit counts, tool args) — not a faithful I/O log, so unusable as-is for training-data extraction the pillar envisions.
- No scientific/data lineage provenance.

## Pillar 8 — Long-term product framing (agent OS for science & engineering)

**Alignment**
- The framing is consistently *written*: `vision.md` = "engineering and scientific research"; ADRs 008/009/010/011 repeatedly name research repos, notebooks, bib files, simulations as targets; [ADR-011] "engine domain-agnostic, discipline layered on top."
- Local-first / ITAR-friendly posture is real via ADRs 002/005/007/012.

**Gaps**
- **Every scientific capability is aspirational text; zero scientific code exists.** The implementation is 100% SWE: `read_file`/`ripgrep` tools, code-oriented FTS, code chunking. No notebook execution, no simulation runner, no data loader, no scientific skills, no `.ipynb`/dataset/bib handling.
- The concrete roadmap through v0.5 is **coding-only** (v0.4 = "local SWE assistant", [ADR-015]). The pillar warns against "retrofitting onto a coding-only platform," yet the current trajectory is exactly that through at least v0.4.
- **No ADR** addresses scientific tool/skill integration, notebook execution, or an experiment/data model — so the primitives (tool registry, chunker, retrieval schema) risk being shaped by coding needs and needing rework for science.

## Existing good ADR patterns (precedent worth following)

The house style is consistent and good: **Status / Context / Decision / Consequences (Positive & Negative) / References**, with heavy cross-linking and pointers to concrete files. Keep it. Standouts:

- **[ADR-014] (incremental index watch)** — the best exemplar. Starts from a **measured problem** ("~280 files, ~5 min"), gives a **primary strategy + explicit fallback**, documents a **storage migration** and a **user upgrade path**, and scopes out-of-scope work. This is the template for future ADRs.
- **[ADR-013] (model tool compatibility)** — excellent for its **empirical matrix**, reproducible **probe methodology**, forward-looking **capability dimensions** table, honest **done/not-done** deliverables, and a rich **Open questions** section.
- **[ADR-007] (remote compute profiles)** — concrete **YAML examples**, and a clean **security separation** (engine key vs backend key) that future policy ADRs should mirror.
- **[ADR-006] (multi-client)** — clear **client taxonomy table** and a **future-API table** that keeps the engine boundary stable.
- **[ADR-011] (governed self-improvement)** — a good model for an intentionally **open-ended "north-star" ADR**: explicit **Intent / Non-goals / Open questions**, "propose don't apply" tenet.

**Weaknesses to fix in the ADR process:** ADRs 001/002/003 are very thin (001 has no Context/Consequences; 003 is two lines). No **ADR template** file, **no dates**, and no **supersession/relationship** discipline. The ratio of Proposed:Accepted is high (10:6) — a large backlog of decided-on-paper-but-unbuilt items; consider a "Deferred" vs "Proposed (committed)" distinction.

## Recommended new ADRs / design docs

Prioritized; each is specific to Annulus's vision, not generic.

1. **ADR-016 — Skill primitive: definition, schema, discovery, and the Tool↔Skill↔Agent boundary.**
   *Problem:* Pillar 4's middle tier does not exist; "skill" lives only as a bullet in [ADR-011]. There is no schema, loader, or storage location for skills, and no stated rule for what belongs in a tool vs a skill vs an agent.
   *Why it matters:* Skills are the packaging unit for the domain workflows that make Annulus more than a coding assistant ("arXiv monitor," "simulation pipeline," "codebase audit"). Without a defined primitive, every domain workflow will be hardcoded into the monolithic `AgentRuntime`, directly violating the separation pillar and blocking Pillar 8.

2. **ADR-017 — Agent identity & configuration model (roles, prompts, memory scope, permissions).**
   *Problem:* There is exactly one hardcoded `AgentRuntime`; [ADR-008]'s swarm assumes configurable agents that do not exist. No way to declare a planner/executor/critic/domain-specialist with its own prompt, tool allowlist, retrieval scope, and permission set.
   *Why it matters:* This is the missing foundation under swarm orchestration ([ADR-008]) and under Pillar 1's "runtime owns agents." Defining it now prevents ADR-008 from being built on sand.

3. **ADR-018 — Permission & capability authorization model.**
   *Problem:* No permission concept exists anywhere in code; the tool sandbox is a single path check. Pillars 1 and 6 both list permissions as first-class, and Pillar 4 wants per-agent permissions.
   *Why it matters:* For ITAR/NASA environments and for `propose_edit`/write tools ([ADR-015]), unbounded autonomous tool execution is unacceptable. Permissions must precede any write-capable tool or multi-agent delegation.

4. **ADR-019 — Data-egress & compliance policy (ITAR / ZDR boundaries).**
   *Problem:* Escalation can ship retrieved workspace content to a frontier API with no governing policy; ZDR is only discussed for inference, never enforced, and never for the context substrate.
   *Why it matters:* The entire premise (local-first for ITAR-restricted NASA machines) fails if there is no enforceable rule for what data may leave the machine and to which endpoints. This is arguably the highest-stakes missing decision.

5. **ADR-020 — Plan-before-act interaction protocol.**
   *Problem:* Pillar 5 has zero coverage — no plan schema, no approve/modify/reject flow, no human checkpoint; the runtime auto-executes up to 8 iterations.
   *Why it matters:* Plan-mode is a named pillar and a core safety/trust mechanism for high-agency CLI and scientific tasks. It must be specified before write/simulation tools arrive.

6. **ADR-021 — Per-project / per-agent policy configuration (`.annulus/config.yaml`).**
   *Problem:* Routing, escalation, permissions, and egress are all global; there is no project-scoped policy object. [ADR-012] hints at a target-repo config but defines no policy surface.
   *Why it matters:* Pillar 3 demands routing "governed by per-project/per-agent policies." This is also the natural home for ADR-018/019 settings and for "this project allows frontier/ZDR."

7. **ADR-022 — Provenance & evaluation data model (separate from observability traces).**
   *Problem:* Traces are truncated observability, not provenance; there is no committed eval/regression harness, and `parent_span_id` is never set so trace trees are flat.
   *Why it matters:* Pillar 7 wants provenance and eval as *core*, feeding SFT/RL and reproducibility. Define a provenance record (full I/O, data + model + tool lineage), wire `parent_span_id`, and specify the eval-suite/export format.

8. **ADR-023 — Scientific workload primitives (notebook execution, data loaders, non-code retrieval).**
   *Problem:* 100% of implemented capability is coding-oriented; no notebook/data/simulation tools or non-code chunking exist, while the roadmap is SWE-only through v0.4.
   *Why it matters:* Pillar 8 explicitly warns against retrofitting science onto a coding platform. An ADR here forces the tool/skill/retrieval primitives to be validated against at least one scientific workload before they ossify.

*(Lower priority / can fold into the above: a Confidence-signal ADR for routing — needed for "escalate when local confidence low," could live inside ADR-021; a Context-substrate-schema ADR — could extend [ADR-009]/[ADR-010].)*

## Top findings (summary)

1. **Skills and multi-agent-as-config are entirely missing** — the intended tool/skill/agent taxonomy is ~1.5 tiers (tools + one monolithic `AgentRuntime`). Biggest structural gap vs Pillar 4.
2. **No permission, plan-before-act, or data-egress model** — three named pillars (5, and the permission facets of 1/6, plus ITAR compliance in 3) have zero code and no ADR. Highest-risk given the ITAR/NASA framing.
3. **Escalation is binary/reactive, not policy-driven** — no confidence signal, no per-project/per-agent policy object; tiered routing ([ADR-007]) is unbuilt.
4. **Observability is strong but not provenance** — trace store + new trace CLI are the best-realized pillar (7), yet `parent_span_id` is unwired (flat trees), attributes are truncated, and there is no eval harness or I/O/provenance record.
5. **The platform is coding-only in practice** — 100% of implemented capability is SWE (`read_file`/`ripgrep`, code FTS); all scientific capability (Pillar 8) is documentation. Roadmap through v0.4 stays SWE-only, risking the exact "retrofit onto coding platform" the vision warns against.
6. **ADR discipline is good and worth codifying** — ADR-014/013/007/006/011 are strong precedents (measured problems, examples, open-questions, non-goals); adopt a template and fix thin early ADRs (001–003), add dates/supersession.
7. **Current branch (trace CLI) is a clean, low-risk Pillar-7 increment** but leaves a latent gap: nesting code is dead until the runtime sets parent spans, and v0.3's "git tools" are not present in this tree (**git tools landed in PR #41 after this review**).
